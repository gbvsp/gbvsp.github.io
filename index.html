Access Certifications

Configured periodic access review campaigns so managers can certify, revoke or escalate user access. Automated the remediation actions triggered by reviewer decisions.

PHASE 01
Certification Campaign Creation
Created manager-based certification campaigns targeting all active identities. Configured campaign parameters — scope (all entitlements), frequency (quarterly), certifier assignment (direct manager), escalation rules (auto-revoke on no response after 14 days).
PHASE 02
Manager Review Workflow
Managers receive email notifications with a link to their certification workitem. Each item shows the identity, their current access and entitlements. Managers can Approve, Revoke or Delegate each access item. Bulk approve options available for clean, low-risk items.
PHASE 03
Automated Remediation
On certification completion, IIQ automatically triggers provisioning plans based on decisions. Revoked entitlements are removed from AD/target systems within the next provisioning cycle. All actions are logged to the audit trail for compliance evidence.
PHASE 04
Reporting & Compliance Evidence
Generated certification completion reports showing decision rates, revocation counts and outstanding items. Reports exported as compliance evidence for audit purposes. Dashboard created to track campaign progress in real time.

02.

Risk Assessment & Register

Conducted a full risk assessment for the EyrieQ LMS platform — identifying assets, threats and vulnerabilities, scoring each risk by likelihood and impact, and defining treatment actions. The risk register is maintained as a living document with quarterly reviews.

PHASE 01
Asset Identification
Catalogued all information assets — user data, LMS content, authentication systems, cloud infrastructure, third-party integrations. Classified each asset by sensitivity (Public / Internal / Confidential / Restricted) and assigned asset owners from the business team.
PHASE 02
Threat & Vulnerability Analysis
For each asset, identified applicable threats (data breach, insider threat, ransomware, account takeover, API vulnerabilities) and mapped existing vulnerabilities. Referenced OWASP Top 10 for web application threats and NIST threat catalogue for broader coverage.
PHASE 03
Risk Scoring (5×5 Matrix)
Scored each risk on a 5-point Likelihood scale (Rare to Almost Certain) × 5-point Impact scale (Negligible to Critical). Calculated inherent risk score and residual risk score after controls. Risks scoring 15+ flagged as High and escalated for immediate treatment planning.
PHASE 04
Risk Treatment & Tracking
For each identified risk, defined a treatment option: Mitigate (implement control), Accept (document and monitor), Transfer (cyber insurance / vendor SLA), or Avoid (stop the activity). Assigned owners and target remediation dates. Tracked in the risk register with status updates.

06.

Post-Quantum Cryptography R&D

Conducted research on Post-Quantum Cryptography (PQC) algorithms being standardised by NIST — assessing their applicability to EyrieQ's LMS platform and recommending a quantum-readiness roadmap.

CRYSTALS-Kyber
Key Encapsulation Mechanism (KEM)
NIST selected KEM for general encryption. Researched for replacing RSA/ECDH in TLS handshakes. Low overhead suitable for LMS API communications.
CRYSTALS-Dilithium
Digital Signature Algorithm
NIST primary digital signature standard. Researched as replacement for RSA/ECDSA signatures in document signing and authentication tokens.
FALCON
Lattice-based Signature
Compact signature sizes compared to Dilithium. Researched for use cases requiring smaller signature footprints in constrained environments.
SPHINCS+
Hash-based Signature
Stateless hash-based scheme. Researched as a conservative backup option given its security relying only on hash function security assumptions.
FINDINGS
Quantum Readiness Recommendations
Recommended a hybrid approach — deploying classical + PQC algorithms in parallel during the transition period to maintain compatibility while building quantum resistance. Key recommendations: (1) Adopt CRYSTALS-Kyber for API key exchange in the next TLS upgrade cycle, (2) Evaluate Dilithium for JWT signing tokens, (3) Conduct crypto-agility audit to identify all hard-coded classical algorithms across the LMS codebase for future migration planning.

arp_spoofer.py
ARP Spoofer (MITM)
Performs ARP poisoning to position itself between a target and the router. Sends crafted ARP packets using Scapy to trick both the victim and gateway into sending traffic through the attacker — enabling full MITM interception.
packet_sniffer.py
Packet Sniffer
Intercepts and filters live network traffic after ARP spoofing. Extracts and displays usernames, passwords, visited URLs and cookies from HTTP traffic in real time using Scapy's sniff() function.
dns_spoofer.py
DNS Spoofer

file_interceptor.py
File Download Interceptor
Intercepts file downloads (.exe, etc.) on the network and replaces them on the fly with a malicious file of our choosing — all without the victim noticing. Uses Netfilterqueue to modify packets mid-stream.
code_injector.py
Code Injector
Injects arbitrary JavaScript or HTML code into HTTP responses intercepted via MITM. Modifies live web pages as they load on the victim's browser — can inject alerts, scripts, or redirect content.
arpspoof_detector.py
ARP Spoof Detector
Defensive tool — monitors ARP packets on the network using Scapy and alerts when it detects inconsistencies between ARP responses and the real gateway MAC, indicating an ongoing ARP spoofing attack.

01.

Abstract

Digital certificates are vital in providing a secure way to authenticate the identity of a web server and the content present in it. Without proper authentication one cannot trust the identity of the issuer or the content — leading to real-world security issues like faking digital certificates and stealing sensitive banking information during transactions. This project identifies the best digital certificate tool by comparing and evaluating OpenSSL and Java Keytool — both used for creating, managing and maintaining keystores. Self-signed certificates were created, configured into an Apache server, deployed in browser trusted authority lists and the communication was secured server-side. Conclusion: OpenSSL is superior to Keytool in terms of library functions and configuration capabilities, while Keytool's primary purpose remains keystoring and certificate management.
02.

PKI & Background

🏛️
Public Key Infrastructure (PKI)
PKI is built on public key encryption and provides digital signature and encryption services. It consists of a CA, certificate library, key backup/restore system, certificate revocation mechanism and an API. The CA is the core — issuing, updating, withdrawing and validating numerical certificates.
🔐
Self-Signed Certificates
Public key certificates not issued by a CA. They are free and used for testing — but since a third party might redirect the connection by faking the certificate, the client cannot be assured of the server's identity. Key size is customisable; CRL and OCSP checks can be done. Suitable for internal organisational use.
🔒
RSA Algorithm
Asymmetric cryptography using public/private key pairs. Security relies on the difficulty of factoring large numbers that are the product of two large primes. Generated using Rabin-Miller primality test. Project used RSA 4096-bit for maximum encryption strength during data transmission.
🔗
TLS/SSL & Cipher Suites
A cipher suite is a set of algorithms securing a network connection. TLS 1.3 is used. TLS 1.2 rejects MD5-signed certificates due to cryptographic weakness. SHA-256 and SHA-512 used throughout the project for hashing. The SSL handshake establishes a secure channel between client and server.
03.

OpenSSL Implementation

OpenSSL is a command-line crypto library implementing TLS/SSL. It was chosen for its platform independence, wide functionality, and use by ~4000 companies worldwide. Key operations below are the exact commands used in the project.

Step 1 — Generate Root CA private key (RSA 4096 + AES-256 passphrase)Kali Linux
# Create directory structure mkdir -p /home/kali/Desktop/sh/root/ca/\{newcert,crl,privatekey,request,cert\} touch ind.txt ind.txt.attr && echo '1000' > ser # Generate root private key — RSA 4096 encrypted with AES-256 openssl genrsa -aes256 -out privatekeys/root.192.168.198.128.key 4096
Step 2 — Self-sign the Root CA certificate (X.509 v3, SHA-512, 3000 days)Kali Linux
openssl req -config open_root.cnf -new -x509 -sha512 -extensions v3_ca -key privatekeys/root.192.168.198.128.key -out cert/root.192.168.198.128.crt -days 3000 -set_serial 0 # Common Name set to: 192.168.198.128 (localhost IP)
Step 3 — Create Intermediate CA CSR and sign with RootKali Linux
# Generate intermediate key + CSR openssl req -config openssl_intermediate.cnf -new -newkey rsa:4096 -keyout intermediate/private/int.192.168.198.128.key -out intermediate/csr/int.192.168.198.128.csr # Root CA signs the intermediate certificate openssl ca -config openssl_root.cnf -extensions v3_intermediate_ca -days 3000 -notext -md sha512 -in intermediate/csr/int.192.168.198.128.csr -out intermediate/cert/int.192.168.198.128.crt
Step 4 — Create certificate chain + Server certificateKali Linux
# Chain intermediate + root into bundle cat intermediate/cert/int.192.168.198.128.crt cert/root.192.168.198.128.crt > intermediate/cert/chain.192.168.198.128.crt # Generate server CSR (RSA 2048, SAN config) openssl req -out intermediate/csr/192.168.198.128.csr -newkey rsa:2048 -nodes -keyout intermediate/privatekey/192.168.198.128.key -config openssl_csr_san.cnf # Intermediate signs the server certificate openssl ca -config openssl_intermediate.cnf -extensions server_cert -days 3000 -notext -md sha512 -in intermediate/csr/192.168.198.128.csr -out intermediate/cert/192.168.198.128.crt # Convert to .pfx for browser upload openssl pkcs12 -inkey intermediate/privatekey/192.168.198.128.key -in intermediate/cert/192.168.198.128.crt -export -out 192.168.198.128.pfx
05.

Java Keytool Implementation

Java Keytool is a key and certificate management utility that stores keys in a .jks keystore. It was configured with RSA algorithm and SHA-384 hash, then validated as a trusted entity in the trust store.

Keytool — Create keystore, generate keypair, import to truststoreWindows / Java
# Generate RSA keypair and self-signed cert in keystore keytool -genkeypair -alias key -keyalg RSA -keystore key.jks # List keystore contents verbosely keytool -list -v -keystore key.jks # Result: SHA384withRSA signature, chain length 1, private key entry # Export certificate to file keytool -export -alias key -keystore key.jks -file certificate.cer # Import into truststore (marks as trusted CA) keytool -import -alias cert -file certificate.cer -keystore trust.jks # Verify trust store keytool -list -v -keystore trust.jks # Result: trustedCertEntry — certificate validated as trusted entity
06.

Browser Deployment Results

After configuring the Apache XAMPP server with the certificate paths (server key, server cert, server chain file), the certificates were uploaded to the trusted authority stores of three browsers. Results revealed significant differences in how browsers handle unregistered domain certificates.

🦊 Mozilla Firefox
⚠️ Warning + Redirect
Certificates uploaded via Settings → Privacy & Security → View Certificates → Trusted Authorities. Root and Intermediate CAs added; server .pfx added to My Certificates. Firefox threw a security warning because the domain is unregistered — but allowed the user to accept the risk and proceed to the website. HTTPS connection was established despite the warning.
🔷 Microsoft Edge
⚠️ Warning — No Redirect
Root and Intermediate CAs imported into the Certificate Authority store; server .pfx added to Your Certificates. Edge flagged the site as not secure due to the unregistered domain but did not redirect. The page was still accessible directly without any browser redirect. Similar to Firefox behaviour but without the redirect option.
🟢 Google Chrome
🚫 Blocked — No Access
Chrome trusted the certificate as approved/signed but did not trust the issuing Certificate Authority (unregistered). Unlike Firefox and Edge, Chrome did not allow the user to bypass the warning or redirect to the website at all. Chrome was the most secure — it fully blocked access to sites with certificates from untrusted CAs.
Key Finding: Browser behaviour shows that Firefox and Edge do not check whether a certificate is from a registered/unregistered domain — a significant security concern. Chrome enforces stricter trust validation, making it the safest browser in this context. The unregistered domain was the root cause of all warnings — registering the domain and obtaining a CA-verified certificate would eliminate these issues.

08.

Conclusion & Recommendations

✅
OpenSSL Wins
OpenSSL is significantly better than Keytool in terms of crypto library functions and configuration capabilities. Supports full 3-tier CA hierarchy, multiple file formats, stronger encryption (AES vs DES) and more hashing options.
🎯
Keytool's Role
Keytool is best suited purely for certificate and key management within a Java keystore. It is not a general-purpose crypto library — its strength is simplicity for storing and organising keys/certs in .jks format.
📌
Scope of Use
Self-signed certificates implemented in this project can only be used on small-scale websites or within an organisation on dedicated web servers. They cannot be used commercially without a registered domain and trusted CA validation.
🔮
Recommendations
Improve certificate lifecycle management. Build a combined portal for generate/store/renew/revoke. Enable timestamping on software tools. Automate CRL monitoring. Register domain and use trusted CA for production deployments.
